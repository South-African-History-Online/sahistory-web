# Classroom 2.0 - HTML presentation format (proposal)

Issue: #436 (child of epic #433)
Status: proposal + working proof of concept
Prototype: `webroot/modules/custom/saho_classroom/prototype/group-areas-act.html`

## 1. Problem

Classroom 2.0 needs to publish teacher-facing presentations. The legacy
practice is PowerPoint / PDF attachments. We want lessons that are:

- viewable in the browser with no download and no plugin,
- printable to PDF for offline classroom use,
- accessible (WCAG 2.1 AA) and responsive on projector, laptop and phone,
- machine-generated by the content agents (#438-443) and rendered by Drupal,
- durable open educational resources, not opaque binary blobs.

The open question this issue derisks: **can a lightweight, self-contained
HTML slide format replace PowerPoint without dragging in a heavy JS library?**

## 2. What the prototype proves

`group-areas-act.html` is a single self-contained file (all CSS and JS inline,
every image inline SVG, no CDN / fonts / external image URLs). It opens from
`file://` with no server and works fully offline. It demonstrates every
capability we needed to see before committing to the approach:

| Requirement | How the prototype meets it |
| --- | --- |
| Arrow / keyboard nav | Arrows, Space, PageUp/Down, Home/End |
| Fullscreen present mode | Fullscreen API, `F` key + Present button |
| Speaker-notes toggle | `S` key + Notes button; per-slide notes panel |
| Clean print-to-PDF | `@media print` renders one slide per page, notes as a handout |
| Fully offline | Zero external requests; inline SVG imagery |
| Responsive | `clamp()` typography, grid columns collapse under 720px, swipe on touch |
| Contrast + a11y | SAHO tokens (AA), semantic sections, `aria-live` slide announcements, focus management, skip link, keyboard-reachable dots |

It weighs ~30 KB, loads instantly, and has no build step or runtime
dependency. The content ("The Group Areas Act", 8 slides) is real Grade 10
History illustrative material with speaker notes, mapped to a CAPS topic.

## 3. Options considered

### Option A - Lightweight custom HTML (recommended)

A tiny hand-rolled engine (~200 lines of vanilla JS + one CSS block), rendered
by a Twig template from a structured Presentation entity.

- **Pros:** no dependency to track for CVEs or Drupal-11 compatibility; total
  control of markup so accessibility and SAHO branding are first-class;
  trivially self-contained and offline; tiny payload aligns with the site's
  Core Web Vitals work; the same structured data can also render as a normal
  scrollable article page for SEO and for JavaScript-disabled users.
- **Cons:** we own the ~200 lines (must maintain nav/print/fullscreen
  ourselves); no built-in speaker-view-on-second-monitor, fragment/step
  animations or transition library out of the box.

### Option B - reveal.js

The de facto HTML-slides library.

- **Pros:** mature, rich feature set (fragments, speaker view with timer,
  auto-animate, Markdown plugin, PDF export mode); large community.
- **Cons:** ~350 KB+ of JS/CSS plus plugins; its offline/self-contained and
  print paths need deliberate configuration; another dependency to patch and
  to keep compatible with our CSP and Drupal aggregation; heavier than the
  whole rest of a lesson page; its default markup and theming would need
  overriding to hit SAHO branding and our exact a11y bar anyway. For our
  needs (linear teacher decks, print-first) most of reveal's surface area is
  unused weight.

### Option C - keep PowerPoint / PDF attachments

- **Cons:** not viewable inline, not responsive, weak accessibility, not
  machine-generable into structured content, opaque to search and to the
  archive mission. Rejected.

## 4. Recommendation

**Adopt Option A - the lightweight custom HTML format.**

Rationale: our decks are linear teacher presentations whose hard requirements
are print-to-PDF, offline use, accessibility and SAHO branding - exactly the
areas where a custom template beats a general-purpose library, and where
reveal.js's extra weight buys features (fragment animations, remote control)
we do not need. Keeping the engine tiny and dependency-free also fits the
site's performance and long-term-maintenance posture. The prototype shows the
full feature set is achievable in ~30 KB with no dependency.

Crucially, generation agents should **not** emit the final HTML. They emit the
**structured slide schema** below; Drupal stores it on a `presentation` entity
and a Twig template renders both an interactive deck (the prototype's markup)
and a plain semantic fallback. This keeps content portable, translatable
(#437), searchable and re-themeable.

If step-by-step fragment animations or a dual-screen presenter view later
become firm requirements, revisit reveal.js as a progressive enhancement over
the same schema - the schema is the durable asset, the renderer is swappable.

## 5. Proposed structured slide schema

Generation agents emit this JSON (or the equivalent Markdown front-matter for
human authors). It is deliberately renderer-agnostic. Field names align with
the existing `saho_classroom` model (`field_classroom_grade`,
`field_classroom_subject`, `field_caps_topic`, `field_classroom_resource_type`).

```json
{
  "schema_version": "1.0",
  "id": "group-areas-act",
  "title": "The Group Areas Act",
  "subject": "History",
  "grade": "10",
  "phase": "FET",
  "caps_topic": "Apartheid legislation in the 1950s",
  "language": "en",
  "duration_minutes": 30,
  "attribution": "South African History Online Classroom",
  "slides": [
    {
      "id": "s1",
      "layout": "title",
      "eyebrow": "Grade 10 History - Apartheid legislation",
      "title": "The Group Areas Act",
      "lead": "How one 1950 law divided South Africa's cities by race.",
      "blocks": [],
      "media": null,
      "notes": "Open by asking what learners know about where people were allowed to live under apartheid."
    },
    {
      "id": "s4",
      "layout": "timeline",
      "eyebrow": "Chronology",
      "title": "From statute to bulldozer",
      "blocks": [
        { "type": "timeline", "events": [
          { "year": "1950", "label": "Act passed" },
          { "year": "1966", "label": "District Six declared white" },
          { "year": "1968-82", "label": "Forced removals" },
          { "year": "1991", "label": "Act repealed" }
        ]}
      ],
      "media": null,
      "notes": "Point out the gap between the 1950 law and the mass removals of 1968-82."
    }
  ]
}
```

### Field reference

Lesson (top level):

| Field | Type | Notes |
| --- | --- | --- |
| `schema_version` | string | Semantic version of this schema. |
| `id` | string | Machine name / URL slug. |
| `title` | string | Lesson title (title slide). |
| `subject` | enum | `History` or `Geography` (`classroom_subject`). |
| `grade` | string | `4`-`12`. |
| `phase` | enum | `Intermediate`, `Senior`, `FET` (`classroom_grade`). |
| `caps_topic` | string | Term from `caps_topic` vocabulary. |
| `language` | string | ISO code; enables the #437 multilingual variants. |
| `duration_minutes` | integer | Estimated teaching time. |
| `attribution` | string | Source / rights line. |
| `slides` | array | Ordered slide objects. |

Slide:

| Field | Type | Notes |
| --- | --- | --- |
| `id` | string | Stable per-slide id (anchors, deep links). |
| `layout` | enum | `title`, `content`, `two-column`, `cards`, `quote`, `image`, `timeline`, `summary`. Drives the Twig sub-template. |
| `eyebrow` | string | Small kicker above the heading. |
| `title` | string | Slide heading. |
| `lead` | string | Optional emphasised intro line. |
| `blocks` | array | Ordered content blocks (below). |
| `media` | object or null | `{ "type": "svg" \| "image", "alt": "...", "data": "..." }`. `data` is inline SVG markup or a data URI so decks stay self-contained. |
| `notes` | string (Markdown) | Speaker notes; rendered in the notes panel and the printed handout. |

Block types inside `blocks`:

| `type` | Shape |
| --- | --- |
| `paragraph` | `{ "type": "paragraph", "text": "Markdown..." }` |
| `list` | `{ "type": "list", "ordered": false, "items": ["...", "..."] }` |
| `card` | `{ "type": "card", "heading": "...", "text": "..." }` |
| `quote` | `{ "type": "quote", "text": "...", "cite": "..." }` |
| `timeline` | `{ "type": "timeline", "events": [ { "year": "...", "label": "..." } ] }` |
| `qa` | `{ "type": "qa", "question": "...", "marks": 4 }` |

### Markdown front-matter equivalent (human authors)

```markdown
---
schema_version: "1.0"
id: group-areas-act
title: The Group Areas Act
subject: History
grade: "10"
phase: FET
caps_topic: Apartheid legislation in the 1950s
language: en
duration_minutes: 30
---

## slide: title
eyebrow: Grade 10 History - Apartheid legislation
title: The Group Areas Act
lead: How one 1950 law divided South Africa's cities by race.
notes: Open by asking what learners already know.

## slide: cards
title: What the Group Areas Act actually did
- card: Divided the map | It let the government reserve any area for one racial group.
- card: Forced removals | Anyone in the "wrong" area could be ordered to move out.
notes: Walk the cards in order - they escalate from lines to demolition.
```

Both forms deserialise to the same internal structure, so agents can emit JSON
while editors can hand-author Markdown.

## 6. How Drupal renders it

1. `presentation` entity stores the schema (a JSON field, or mapped to native
   Paragraph/field storage for full editability).
2. `saho-classroom-presentation.html.twig` iterates `slides` and includes a
   per-`layout` sub-template that produces the same markup as the prototype.
3. The interactive engine (the prototype's inline CSS/JS, moved into the
   `saho_classroom` theme asset library) is attached to the deck wrapper.
4. A `?view=print` / no-JS path renders the same slides as one linear,
   scrollable, semantic article - good for SEO, screen readers and printing.

  (respecting the site CSP - keep it self-contained, no inline eval).

  `language`; media/SVG shared across translations.
